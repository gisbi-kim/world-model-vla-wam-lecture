World Model, VLA, WAM: Embodied Robot Intelligence의 현재 구조

이 강의문은 최근 World Model 논의에서 제기된 renderer, simulator, planner의 기능적 구분을 출발점으로 삼아, Vision-Language-Action 모델과 World Action Model의 흐름을 하나의 체계로 정리한다. 목적은 용어의 나열이 아니라, 궁극적으로 물리 세계에서 작동하는 embodied robot을 만들기 위해 어떤 구성 요소가 필요하며, 왜 단일한 대형 모델만으로는 충분하지 않은지를 설명하는 데 있다.

1. 문제의 출발점

최근 인공지능 연구에서 world model이라는 말은 매우 넓게 쓰인다. 어떤 경우에는 비디오 생성 모델을 가리키고, 어떤 경우에는 강화학습에서 환경의 동역학을 예측하는 내부 모델을 뜻하며, 또 다른 경우에는 로봇이 실제 세계에서 행동하기 위해 필요한 물리적 이해 전체를 뜻한다. 이 말의 범위가 넓어질수록, 논의는 빠르게 모호해진다.

Fei-Fei Li와 World Labs 팀의 글 A Functional Taxonomy of World Models 는 이 혼란을 정리하기 위해 world model을 세 기능으로 나눈다[1]. 그 세 기능은 renderer, simulator, planner이다. 이 구분은 특정 아키텍처의 이름이 아니다. 오히려 한 지능 시스템이 세계와 관계를 맺기 위해 수행해야 하는 기능적 역할의 분해이다.

핵심 명제. World model 논의의 핵심은 “그럴듯한 장면을 생성하는가”가 아니라, “세계의 상태를 표현하고, 행동의 결과를 예측하며, 그 예측을 근거로 다음 행동을 선택할 수 있는가”에 있다.

2. 세 기능: Renderer, Simulator, Planner

세 기능은 서로 대체 관계가 아니다. 하나의 embodied intelligence가 완성되려면 세 기능이 닫힌 고리(closed loop) 안에서 연결되어야 한다. 하지만 현재 연구와 산업 제품은 대개 이 중 일부 기능을 강하게 수행하고, 다른 기능은 약하게 수행한다.

Renderer

세계가 어떻게 보이는가

Renderer는 내부 표현이나 조건 입력으로부터 관측 가능한 장면을 만든다. 이미지, 비디오, 3D view, Gaussian splat, mesh 등이 여기에 속한다. Sora, Veo, Marble과 같은 생성 시스템은 이 기능을 강하게 보여준다[8][9][10].

Simulator

세계가 어떻게 변하는가

Simulator는 현재 상태와 행동이 주어졌을 때 다음 상태, 다음 관측, 보상, 실패 가능성 등을 예측한다. Genie, Cosmos, DreamerV3는 이 기능을 중심으로 이해할 수 있다[11][13][14].

Planner

무엇을 해야 하는가

Planner는 목표를 달성하기 위해 행동을 선택한다. 로봇 분야에서 RT-2, OpenVLA, π₀와 같은 VLA 모델은 대체로 이 위치에 놓인다 [5][6][7].

이 구분을 비디오 생성 모델에 적용하면 중요한 경계가 보인다. 어떤 모델이 현실적인 영상을 만든다고 해서 곧바로 simulator가 되는 것은 아니다. 시각적 일관성은 renderer의 성취일 수 있다. Simulator라고 부르려면, 행동이 개입되었을 때 세계가 어떻게 바뀌는지를 안정적으로 예측해야 한다. 따라서 Sora나 Veo류의 모델은 world simulator를 향한 중요한 징후를 보이지만, 기본 위치는 여전히 renderer에 가깝다[8][9].

3. VLA의 위치

Vision-Language-Action, 즉 VLA 모델은 시각 관측과 언어 지시를 받아 로봇 행동을 출력하는 모델이다. 가장 단순한 도식은 다음과 같다.

image / observation + language instruction + robot state -> action

RT-2는 웹 규모의 vision-language 사전학습을 로봇 제어로 옮기려 한 대표적 사례이며[5], OpenVLA는 공개된 7B 규모 VLA 모델로 다양한 로봇 demonstration을 학습해 일반 조작 능력을 보였다[6]. π₀는 VLM 위에 flow matching 기반 action expert를 얹어 연속적이고 정밀한 로봇 행동을 생성하려 했다[7].

VLA의 강점은 의미 일반화(semantic generalization)에 있다. 물체 이름, 지시문, 장면의 의미를 이해하고 행동으로 변환하는 능력은 이전의 좁은 imitation policy보다 훨씬 강하다. 그러나 VLA는 대체로 reactive observation-to-action mapping이다. 즉, 현재 관측에서 곧바로 행동으로 간다. 이 구조는 다음과 같은 문제를 남긴다.

문제 설명 필요한 보완
물리적 선견성 부족 행동 뒤의 접촉, 미끄러짐, 충돌, 실패를 명시적으로 상상하지 못한다. Action-conditioned world prediction
장기 과제의 누적 오류 긴 작업에서는 작은 오차가 누적되어 실패한다. Temporal abstraction, rollout evaluation
Embodiment gap 사람 손, 다른 로봇 팔, 휴머노이드의 행동 공간이 서로 다르다. Retargeting, shared action representation
데이터 병목 로봇 demonstration은 비싸고 느리며 환경 다양성이 제한된다. Human video, simulation, auto-labeling

2024년 VLA 서베이는 VLA 연구를 하나의 분야로 정리했고[2], 이후 action tokenization 관점의 서베이는 로봇 행동을 어떤 단위와 형식으로 모델에 넣고 뽑을 것인가를 중요한 쟁점으로 부각했다[4]. 2026년의 데이터 중심 서베이는 여기서 한 걸음 더 나아가, VLA의 다음 진전은 모델 구조만이 아니라 데이터셋, 벤치마크, 데이터 엔진의 공동 설계에 달려 있다고 주장한다[3].

4. WAM의 등장

World Action Model, 즉 WAM은 VLA와 world model의 결합을 가리키는 최근 용어이다. 간단히 말하면, VLA가 “보고 곧바로 행동하는” 모델이라면, WAM은 “행동 전후의 세계 변화를 모델링하면서 행동하는” 모델이다.

observation + language + world prediction <-> action generation

2026년의 WAM 서베이는 WAM을 “predictive world modeling과 action generation을 통합하는 embodied foundation model”로 정의한다 [15]. 이 정의에서 중요한 점은 WAM이 단순히 world model을 옆에 붙인 VLA가 아니라는 것이다. WAM은 미래 상태와 행동을 함께 모델링한다. 말하자면 planner가 simulator를 내장하기 시작한 형태이다.

Cascaded WAM과 Joint WAM

WAM 서베이는 WAM을 크게 두 계열로 나눈다[15]. 첫째는 Cascaded WAM이다. 여기서는 world model이 먼저 미래 관측이나 상태를 생성하고, 별도의 action model이 그 미래를 바탕으로 행동을 선택한다. 둘째는 Joint WAM이다. 여기서는 미래 상태와 행동이 하나의 모델 안에서 함께 생성되거나 함께 정렬된다.

구조 장점 한계
Cascaded WAM world prediction과 action decoding의 역할이 명확하다. 추론 시간이 길고, 생성된 미래와 실제 행동 사이의 정합성이 깨질 수 있다.
Joint WAM 미래 상태와 행동을 공동 분포로 다루므로 video-action alignment가 강해질 수 있다. 학습 안정성, 데이터 요구량, 평가 기준이 복잡하다.

DreamZero는 WAM을 zero-shot policy로 제안하며, 비디오 diffusion backbone 위에서 미래 세계 상태와 행동을 함께 학습하려 한다 [16]. Fast-WAM은 여기서 한 가지 중요한 질문을 던진다. 로봇이 실제로 행동할 때마다 미래 비디오를 생성해야 하는가? 이 논문은 test-time future imagination 자체보다, 학습 중 video modeling objective가 representation을 형성하는 효과가 더 중요할 수 있음을 보인다 [17]. OA-WAM은 물체별 slot과 identity를 통해 조작 대상의 addressability를 높이려 한다[18].

정리. WAM은 Fei-Fei Li의 세 기능으로 보면 simulator와 planner의 결합이다. 다만 비디오 생성 backbone을 사용할 경우 renderer 성분도 포함한다. 따라서 WAM은 세 부류 중 하나가 아니라, 세 기능이 서로 결합되는 현재 연구 경향을 보여주는 이름이다.

5. 2026년 이후 서베이의 공통 주장

최근 서베이들은 서로 다른 용어를 쓰지만, 공통된 방향을 가리킨다. 그 방향은 “더 큰 VLA를 만들면 로봇 지능이 완성된다”가 아니다. 오히려 VLA, world model, reward model, embodiment interface, data engine이 함께 구성되어야 한다는 쪽이다.

5.1 World model은 policy를 위한 예측 장치이다

World Model for Robot Learning 서베이는 world model을 policy learning, planning, simulation, evaluation, data generation, robotic video generation의 중심 도구로 본다[12]. 여기서 world model은 단순한 생성기가 아니라, 로봇이 행동의 결과를 계산하고, 실제 시행착오를 줄이며, 실패를 미리 평가하게 하는 예측 장치이다.

5.2 데이터 인프라가 모델 구조만큼 중요하다

Vision-Language-Action in Robotics 서베이는 VLA 연구를 datasets, benchmarks, data engines라는 세 축으로 정리한다[3]. 이 관점에서는 모델 크기보다 중요한 문제가 있다. 로봇이 학습할 수 있는 형태로 물리 세계의 데이터를 어떻게 바꾸는가이다.

인터넷 비디오, egocentric video, 사람의 동작, 시뮬레이션 rollout은 모두 풍부한 행동 정보를 포함한다. 그러나 대부분은 로봇의 joint command, gripper state, reward, task success label을 갖고 있지 않다. 따라서 다음 단계의 핵심은 데이터를 더 많이 모으는 것만이 아니라, 비정형 행동 데이터를 로봇 학습 가능한 supervision으로 변환하는 것이다.

5.3 VLA와 world model만으로도 부족하다

Robots Need More than VLA and World Models는 제목 그대로 강한 주장을 편다 [20]. 이 논문은 일반 로봇 지능의 병목이 policy learning 하나가 아니라고 본다. 세계에는 방대한 비정형 행동 데이터가 있지만, 그것을 로봇에게 쓸 수 있는 supervision으로 바꾸는 mechanism이 부족하다는 것이다.

이 논문은 다음 네 가지 interface를 제안한다.

Interface 역할 해결하려는 병목
Data interface 비정형 행동 데이터를 자동 라벨링하고 로봇 학습 데이터로 변환한다. Robot demonstration의 비용과 규모 한계
Embodiment interface 사람 동작이나 다른 로봇의 행동을 목표 로봇의 행동 공간으로 retargeting한다. Embodiment gap
World-model interface 물리적으로 grounded된 3D reasoning과 미래 예측을 제공한다. Reactive policy의 선견성 부족
Reward interface 비디오와 언어에서 task progress, success, failure, safety를 추론한다. Long-horizon learning과 self-improvement의 평가 문제

6. Embodied Robot을 위한 연구 의제

지금의 흐름을 종합하면, 궁극적 embodied robot은 하나의 거대한 end-to-end network를 단순히 키운다고 만들어지지 않는다. 필요한 것은 물리 세계와 지속적으로 접속하는 학습 체계이다. 이 체계는 적어도 다음 여섯 조건을 만족해야 한다.

6.1 의미 이해와 행동 생성을 결합해야 한다

로봇은 언어 지시를 이해해야 한다. 물체 이름, 장소, 관계, 암묵적 목표를 해석해야 한다. 이 점에서 VLA는 필수이다. 그러나 VLA는 planner 또는 policy로 이해해야 한다. 그것은 세계가 어떻게 변할지에 대한 충분한 모델을 자동으로 포함하지 않는다.

6.2 행동 결과를 예측해야 한다

로봇이 컵을 집을 때 중요한 것은 “컵이라는 단어를 안다”는 사실이 아니다. 어느 방향에서 접근하면 미끄러지는지, 어느 힘을 넘으면 넘어지는지, 장애물과 충돌하지 않는지, 실패했을 때 어떤 회복 행동이 가능한지를 알아야 한다. 이 영역이 simulator의 영역이다.

6.3 시각적 생성은 물리적 예측과 구별되어야 한다

현실적인 비디오를 생성하는 능력은 중요하다. 그것은 데이터 생성, future observation, human inspection, virtual training에 쓰인다. 그러나 시각적으로 그럴듯한 장면과 물리적으로 올바른 상태 변화는 다르다. Renderer의 성공이 곧 simulator의 성공은 아니다.

6.4 데이터는 로봇 demonstration 바깥에서 와야 한다

로봇 teleoperation만으로는 충분한 규모와 다양성을 얻기 어렵다. 인간 행동 비디오, 1인칭 영상, 시뮬레이션, 실패 로그, 배포 후 상호작용이 모두 학습 재료가 되어야 한다. 문제는 이것들이 곧바로 행동 라벨이 아니라는 점이다. 따라서 data interface와 reward interface가 필요하다.

6.5 Embodiment를 추상화해야 한다

사람의 손, Franka arm, ALOHA, 휴머노이드, 이동 조작 로봇은 서로 다른 신체를 가진다. 동일한 “문을 연다”라는 과제도 action space는 완전히 다르다. 그러므로 로봇 지능은 특정 joint trajectory에만 묶여서는 안 된다. 접촉, affordance, object-centric state, skill primitive, retargetable motion representation이 필요하다.

6.6 배포 후 자기개선 루프가 있어야 한다

실제 로봇은 실험실 밖에서 실패한다. 실패를 수집하고, world model 안에서 counterfactual rollout을 만들고, reward model로 성공 가능성을 평가하며, policy를 다시 학습하는 루프가 있어야 한다. 이 루프가 없으면 로봇은 정적인 벤치마크에서는 좋아 보여도 실제 환경의 분포 변화에 약하다.

7. 가능한 시스템 구조

현재 문헌을 바탕으로 가장 설득력 있는 구조는 단일 계층의 monolithic agent가 아니라, 계층적이고 폐루프적인 robot foundation system이다. 이 구조는 다음과 같이 표현할 수 있다.

1. Semantic Layer 언어 지시, 장면 이해, 목표 분해를 담당한다. VLM/VLA가 이 층의 중심이다.
2. World Layer 상태, 물체, 접촉, 미래 관측을 예측한다. World model 또는 WAM이 이 층을 담당한다.
3. Action Layer 행동 chunk, skill primitive, continuous control을 생성한다. VLA action head, diffusion policy, flow policy가 여기에 놓인다.
4. Control Layer 실제 하드웨어 안정성, collision avoidance, force control, safety constraint를 처리한다.

여기에 data engine과 deployment loop가 붙는다. 로봇은 실제 환경에서 실패와 성공을 기록하고, 그 기록은 다시 reward model, world model, policy 학습으로 들어간다. 이 점에서 embodied robot은 한 번 학습된 모델이라기보다, 세계와 계속 상호작용하며 갱신되는 시스템에 가깝다.

최종 정리. Renderer는 세계를 보이게 한다. Simulator는 세계의 변화를 예측한다. Planner는 행동을 고른다. VLA는 주로 planner이다. World model은 주로 simulator이다. WAM은 이 둘을 결합하려는 현재의 대표적 흐름이다. 궁극적 embodied robot은 여기에 data interface, embodiment interface, reward interface, low-level control, deployment feedback loop를 더해야 한다.
